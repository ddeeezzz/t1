## 日语歌词对齐与噪声清洗修复计划（jieranduhuo）

### Summary
- 目标：解决 4 类问题  
1. 一句歌词跨多个片段导致后续片段显示 `<无>`  
2. 占位图出现“歌词：”或“歌詞”等无意义文本  
3. 开头未开唱出现 `era/ぺけ` 幻觉词  
4. 尾段 `lalala/dadada` 被识别成乱码
- 已锁定策略：  
1. 人声段内“歌词句优先”（同一句不再被切碎）  
2. 三态清洗（正常歌词 / `[未识别歌词]` / 吟唱）  
3. `configs/jieranduhuo.json` 强制 `funasr_language=ja`

### Key Changes
- 模块 A（歌词识别与分段）  
1. ASR 原始片段保留置信相关信号（`confidence/score`）并补齐 `no_speech_prob` 默认值，供后续“疑似有人声”判断。  
2. 在清洗层输出三态：  
   - 可靠文本：原样保留。  
   - 占位词或低置信且疑似有人声：统一转为 `[未识别歌词]`。  
   - vocable（`lalala/dadada` 类）：归一为“吟唱”。  
   - 明显噪声（短拉丁幻觉词、高无声概率片段）：直接过滤。  
3. 在人声段应用“歌词句优先”分段：每个歌词句按其 `start_time/end_time` 形成单独最小单元，禁止一句歌词被拆到多个视觉片段。  
4. 增加防乱切稳定化：  
   - 句间微小空档（<=0.35s）不单独切镜头。  
   - 人声区非歌词碎片（<0.8s）自动合并到邻段。  
5. 对器乐段继续保留节拍/能量驱动切分，避免全局退化。  
6. 分段完成后再挂载 `lyric_units`，确保 `segment_id -> lyric_units` 一一对应（不再出现“后续切片有画面但歌词 `<无>`”）。  
7. `configs/jieranduhuo.json` 的 `module_a.funasr_language` 从 `auto` 改为 `ja`（该曲定向优化）。

- 模块 B/C（展示一致性）  
1. B 端继续输出 `lyric_text/lyric_units`，并采用三态优先级聚合：正常歌词 > `[未识别歌词]` > `吟唱` > 空。  
2. C 端占位图按三态展示：  
   - 有歌词文本：`歌词：{文本}`  
   - 不可靠但有人声：`歌词：[未识别歌词]`  
   - 无人声：`歌词：<无>`  
3. 旧 `module_b_output.json` 仍可读取（`get("lyric_text","")` / `get("lyric_units",[])`）。

- 文档与说明  
1. 在现有配置文档中补充：`jieranduhuo` 为日语样例，推荐 `funasr_language=ja`。  
2. 在歌词链路说明中明确“人声段歌词句优先”与“吟唱归一”规则。

### Test Plan
- 单元测试（模块 A）  
1. 给定跨片段歌词样本，断言输出分段不会把同一句拆成多个 `segment_id`。  
2. 给定占位词（`歌詞/lyrics`）样本，断言输出 `[未识别歌词]`。  
3. 给定 `lalala/dadada` 样本，断言归一为“吟唱”。  
4. 给定低置信 + 低 `no_speech_prob` 样本，断言输出 `[未识别歌词]`。  
5. 给定 `funasr_language=ja`，断言 `generate(language='ja')` 被调用。

- 集成测试（A→B→C）  
1. 构造“句子跨原切点”的输入，断言 B 每个歌词句只出现在一个 shot。  
2. 断言 C 占位图三态展示正确：正常文本 / `[未识别歌词]` / `<无>`。  
3. 断言有 vocable 的片段显示“吟唱”。  
4. 断言句间微小空档与人声短碎片不会触发闪切。

- 端到端回归（jieranduhuo）  
1. 运行 `configs/jieranduhuo.json`，检查 A/B/C/D 均为 `done`。  
2. 抽检开头片段不再出现 `era/ぺけ`；尾段 vocable 显示为“吟唱”或被合理过滤。  
3. 抽检“已切片但歌词 `<无>`”问题消失（句级对齐生效）。

### Assumptions
- “歌词句优先”仅在人声段生效；器乐段仍可按节拍细切。  
- `[未识别歌词]` 是人声存在但转写不可靠的唯一占位文案。  
- 噪声过滤优先级高于占位显示（明显幻觉词直接丢弃）。  
- 本轮先对 `jieranduhuo` 落地 `ja`，其他歌曲继续按各自配置决定语言策略。
