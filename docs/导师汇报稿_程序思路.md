# 导师汇报稿（程序思路版）

> 适用场景：周会汇报、中期检查、答辩演示。  
> 使用方式：可直接照读，也可按“3分钟版”做精简汇报。

---

## 一、项目一句话说明

我们这个 MVP 的核心目标是：  
**输入一段音频，自动生成一段与音乐结构和节拍对齐的短视频，并且支持任务中断后从断点继续运行。**

---

## 二、为什么这样设计（总体思路）

我们没有一开始就追求“最强生成效果”，而是先保证工程链路稳定，具体采用了三条原则：

1. **先打通流程，再优化模型。**  
先用 Mock 方式跑通 A→B→C→D→E，全流程可验证，后续再替换真实模型。

2. **先做结构约束，再做视觉表达。**  
先从音频里拿到时间结构，再让画面去跟随时间轴，避免“画得很好但卡点不准”。

3. **先做可恢复，再做高复杂度。**  
任务状态全部入库，任何模块失败后可以从断点恢复，不必整条流水线重跑。

---

## 三、程序架构（A→B→C→D→E）

### 模块A：音乐理解（时间轴建立）

- 读取音频时长，构建宏观段落（segments）和节拍点（beats）。
- 对人声段采用“歌词句优先”：一句歌词对应一个最小视觉单元，不在句内切分。
- 对 Whisper 结果做清洗：过滤噪声词（如 `歌詞/lyrics`），将 `lalala/dadada` 归一为“吟唱”。
- 生成能量特征（energy_features）作为后续镜头决策依据。
- 输出 `module_a_output.json`。

一句话：**A 模块负责把“音频”变成“可调度的时间结构”。**

### 模块B：分镜脚本生成（规则/可替换）

- 读取 A 的结果，生成分镜数组（每段起止时间、场景描述、运镜、转场）。
- 当前默认是规则生成（Mock），并预留了 LLM 生成器接口。
- 输出 `module_b_output.json`。

一句话：**B 模块负责把“时间结构”变成“可执行的画面脚本”。**

### 模块C：关键帧生成（占位/可替换）

- 读取 B 的分镜，生成关键帧占位图及帧清单。
- 目前默认是 Pillow 占位图，已预留 diffusion 接口。
- 输出 `module_c_output.json` + `frames/*.png`。

一句话：**C 模块负责把“脚本”变成“可拼接的视觉素材”。**

### 模块D：视频合成（真实执行）

- 根据 C 的帧清单构建 FFmpeg concat 文件。
- 将关键帧按时长拼接并混入原始音轨，输出 MP4。
- 输出 `final_output.mp4`。

一句话：**D 模块负责把“素材”变成“最终视频”。**

### 模块E：状态管理（SQLite）

- 管理任务状态与模块状态：`pending/running/done/failed`。
- 记录每个模块产物路径、错误信息、时间戳。
- 支持 `resume` 从第一个非 done 模块继续执行。

一句话：**E 模块保证流程可追踪、可恢复、可调试。**

---

## 四、数据是怎么流动的

程序核心数据链路如下：

1. `module_a_output.json`  
包含 `segments / beats / lyric_units / energy_features`。

2. `module_b_output.json`  
每个 shot 至少包含：  
`shot_id, start_time, end_time, scene_desc, image_prompt, camera_motion, transition`。
歌词增强版本会额外包含：`lyric_text, lyric_units`。  
其中 `lyric_units` 透传模块 A 的歌词时间单元（按小段落聚合、按时间排序），`lyric_text` 用于占位图展示。  
为兼容历史任务恢复，旧版无歌词字段的 `module_b_output.json` 仍可被模块 C 继续消费。

3. `module_c_output.json`  
记录每个 shot 对应的 `frame_path` 与 `duration`，用于 D 模块拼接。

4. `final_output.mp4`  
最终可播放视频。

一句话：**每个模块只读上游 JSON，只写本模块 JSON，边界清晰。**

---

## 五、命令入口与调试方式

系统目前有三个命令入口：

1. `run`：全链路执行  
2. `resume`：断点恢复  
3. `run-module`：单模块调试

这三个入口覆盖了“正常运行、故障恢复、模块联调”三类常见研发场景。

## 六、当前 MVP 已达成结果

当前版本已经可以稳定完成：

1. 输入 `resources/juebieshu20s.mp3` 自动生成 20 秒样片；
2. A/B/C/D 模块状态完整入库；
3. 失败后可从断点恢复，不重跑已完成模块；
4. 模块契约有校验，测试可自动回归；
5. B/C 已预留“真模型替换点”，后续可平滑升级。

## 七、关键工程取舍（导师常问点）

### 取舍1：为什么先用 Mock，不直接上大模型？

因为我们先要验证工程链路正确性。  
如果流程和状态机还不稳定，直接上模型只会放大不确定性，不利于定位问题。

### 取舍2：为什么强调 JSON 契约？

因为模块是松耦合协作。  
契约稳定，才能支持并行开发、单模块替换、后续模型升级。

### 取舍3：为什么要做 SQLite 状态机？

生成任务可能耗时长、依赖多，失败是常态。  
状态机可以把失败成本从“整条重跑”降低为“断点续跑”。

## 八、下一阶段计划（从 MVP 到增强版）

1. 将 B 的 `LlmScriptGenerator` 接入真实 LLM。  
2. 将 C 的 `DiffusionFrameGenerator` 接入真实扩散模型。  
3. 在 D 模块补充更细的转场策略（如 flash）。  
4. 增加质量评估指标（卡点误差、镜头连贯性、风格一致性）。

## 九、3分钟汇报精简版（可直接口述）

老师好，我们现在完成的是一个可运行的 MVP：输入音频后，系统可以自动生成一段卡点视频，并支持断点恢复。  
技术上我们采用 A 到 E 五模块：A 做音频时间结构，B 做分镜脚本，C 生成关键帧，D 用 FFmpeg 合成视频，E 用 SQLite 管状态。  
我们有三个命令入口：`run`、`resume`、`run-module`，分别对应全链路、故障恢复和单模块调试。  
当前版本重点是工程可用性，先用 Mock 跑通流程，同时预留了真实 LLM 和扩散模型的替换接口。  
所以当前成果是：链路可跑、状态可追踪、失败可恢复，下一步就是把 B/C 的占位实现替换成真实模型，提升画面质量和语义表现。
