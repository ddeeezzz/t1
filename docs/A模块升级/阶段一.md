模块 A 歌词链路升级需求

## 阶段一

### 1. 目标

在模块 A 中新增“音乐指纹识别 -> 查询歌曲元数据 -> 下载同步歌词 LRC”的歌词主链。
当主链成功获取到可用的同步歌词时，优先使用该链路产出的歌词时间戳结果，替代当前直接使用 FunASR 作为歌词主结果的方案。

### 2. 方案决策

1. 指纹识别主链采用：
- AcoustID + Chromaprint

2. 同步歌词主链采用：
- LRCLIB

3. 歌词兜底方案采用：
- 保留 FunASR 作为兜底链路

4. 查询条件采用严格匹配：
- artist + title + duration

说明：
- 当前阶段不采用“宽松匹配优先”的策略。
- 这样做的目标是尽量降低拉取到错误版本歌词或错误时间轴的概率，尽可能减小 LRC 时间轴误差。

### 3. 主链执行要求

模块 A 的歌词获取主链调整为：

1. 对输入音频执行音乐指纹识别。
2. 通过 AcoustID 获取歌曲识别结果。
3. 使用 artist + title + duration 查询 LRCLIB。
4. 当 LRCLIB 返回同步歌词（带时间戳的 LRC）时，优先使用该结果进入当前歌词主链。
5. 当前阶段默认直接使用 LRCLIB 返回的同步歌词时间戳，不将 beat 吸附作为主链必选步骤。
6. 只有当 LRCLIB 未返回可用同步歌词时，才进入 FunASR 兜底流。

### 4. 时间戳处理策略

当前阶段默认直接使用 LRCLIB 返回的同步歌词时间戳作为歌词主链结果。

这样定义的原因是：

- 根据已有经验，大多数歌词时间戳本身已经足够准确；
- 当前阶段优先完成“指纹识别 + 同步歌词主链 + FunASR 兜底”的主流程落地；
- 不应在第一阶段就把 beat 吸附作为强制门槛，以免扩大实现复杂度并引入额外误差。

当前阶段约束：

- 允许将 LRCLIB 返回的同步歌词时间戳直接作为当前主链输出；
- 不要求在当前版本中强制执行 snap to nearest beat；
- 只要 LRCLIB 返回的是可用同步歌词，就可以作为当前下游 Seg 切分的时间基准之一。

### 5. 主链与兜底链的判定规则

1. 当 LRCLIB 返回可用同步歌词时：
- 使用 LRC 主链结果；
- 替代当前 FunASR 直接产出的歌词主结果。

2. 当 LRCLIB 仅返回纯文本歌词、没有同步时间戳时：
- 不使用该结果进入当前主链；
- 直接走 FunASR 兜底流。

3. 当 AcoustID 识别失败、歌曲匹配失败、LRCLIB 查询失败，或 LRC 内容不可用时：
- 统一走 FunASR 兜底流。

### 6. 实现边界说明

- 本阶段升级的重点是“新增 LRC 主链，并在当前阶段直接使用可用同步歌词进入下游”；
- FunASR 当前仍保留，职责调整为兜底链；
- 如果主链失败，不应阻塞模块 A 继续完成，应回退到现有 FunASR 兜底能力。

### 7. 接口契约与字段流转

本节只描述阶段一必需的字段契约，目标是明确每一个字段如何从外部服务进入模块 A，再如何进入现有 `ModuleAOutput`。

说明：

- 阶段一不修改 `ModuleAOutput` 最终契约；
- 阶段一新增的主要是“主链内部中间结构”和“来源判定字段”；
- 下游模块 B/C/D 继续只消费现有 `module_a_output.json`，不直接读取 AcoustID 或 LRCLIB 原始返回。

#### 7.1 字段流转总览

阶段一字段流转顺序如下：

1. 输入音频 `audio_path`
2. 音乐指纹识别结果
3. AcoustID 查询结果
4. LRCLIB 查询结果
5. 内部标准化 `lyric_sentence_units`
6. `clean_lyric_units()` 清洗结果
7. `attach_lyrics_to_segments()` 挂载结果
8. 输出到 `ModuleAOutput.lyric_units`

其中：

- `big_segments` 与 `beats` 仍然来自现有 allin1 链路；
- `segments` 与 `energy_features` 仍然来自现有算法层；
- 本次升级只替换“歌词来源”，不改动现有节拍和分段主链。

#### 7.2 外部识别输入契约

主链输入只使用现有模块 A 已有输入，不新增用户侧必填参数。

```json
{
  "task_id": "string",
  "audio_path": "string",
  "duration_seconds": 123.45
}
```

字段说明：

- `task_id`
  - 来源：现有 `RuntimeContext.task_id`
  - 去向：最终继续写入 `ModuleAOutput.task_id`

- `audio_path`
  - 来源：现有 `RuntimeContext.audio_path`
  - 去向：
    - 传给指纹识别链路
    - 传给 FunASR 兜底链路
    - 最终继续写入 `ModuleAOutput.audio_path`

- `duration_seconds`
  - 来源：现有 `probe_audio_duration()` 结果
  - 去向：
    - 作为 AcoustID/LRCLIB 严格匹配辅助字段
    - 不单独写入 `ModuleAOutput`

#### 7.3 指纹识别内部标准化契约

阶段一建议在模块 A 感知层内部新增“指纹识别标准化结果”结构，供后续 AcoustID 查询使用。

```json
{
  "fingerprint": "string",
  "duration_seconds": 123.45,
  "fingerprint_engine": "chromaprint",
  "status": "ok|failed",
  "error": "string"
}
```

字段流转：

- `fingerprint`
  - 来源：Chromaprint/fpcalc 生成结果
  - 去向：传给 AcoustID 查询

- `duration_seconds`
  - 来源：指纹工具输出或模块 A 已探测时长
  - 去向：传给 AcoustID 查询，并参与后续 LRCLIB 严格匹配

- `fingerprint_engine`
  - 来源：固定值 `chromaprint`
  - 去向：仅用于内部日志和调试产物

- `status`
  - 来源：指纹步骤执行结果
  - 去向：驱动“继续主链”或“回退 FunASR”

- `error`
  - 来源：异常信息或失败原因
  - 去向：仅用于日志和调试产物，不进入最终 `ModuleAOutput`

#### 7.4 AcoustID 查询标准化契约

阶段一建议不要把 AcoustID 原始响应直接向下游透传，而是在模块 A 内部先归一化为一个最小可用结构。

```json
{
  "status": "ok|no_match|failed",
  "artist": "string",
  "title": "string",
  "duration_seconds": 123.45,
  "score": 0.98,
  "acoustid_id": "string",
  "recording_id": "string",
  "raw_candidates": []
}
```

字段流转：

- `status`
  - 来源：AcoustID 查询结果判定
  - 去向：决定是否继续查询 LRCLIB，或直接回退 FunASR

- `artist`
  - 来源：AcoustID 命中的主艺人名
  - 去向：作为 LRCLIB 查询键的一部分

- `title`
  - 来源：AcoustID 命中的歌曲名
  - 去向：作为 LRCLIB 查询键的一部分

- `duration_seconds`
  - 来源：AcoustID 命中记录时长或本地音频时长
  - 去向：作为 LRCLIB 查询键的一部分

- `score`
  - 来源：AcoustID 匹配分数
  - 去向：仅用于内部筛选、日志与调试产物

- `acoustid_id`
  - 来源：AcoustID 结果主键
  - 去向：仅用于调试追踪

- `recording_id`
  - 来源：MusicBrainz recording id 或等价标识
  - 去向：仅用于调试追踪

- `raw_candidates`
  - 来源：AcoustID 候选列表
  - 去向：仅用于调试产物，不传给下游模块

#### 7.5 LRCLIB 查询标准化契约

阶段一建议将 LRCLIB 返回统一规范成以下内部结构，避免后续代码直接依赖外部 API 字段名。

```json
{
  "status": "synced|plain|not_found|failed",
  "artist": "string",
  "title": "string",
  "duration_seconds": 123.45,
  "plain_lyrics": "string",
  "synced_lyrics": "string",
  "provider": "lrclib",
  "provider_id": "string",
  "error": "string"
}
```

字段流转：

- `status`
  - 来源：LRCLIB 返回结果类型判定
  - 去向：
    - `synced`：进入 LRC 主链解析
    - `plain`：当前阶段直接回退 FunASR
    - `not_found|failed`：当前阶段直接回退 FunASR

- `artist`
  - 来源：LRCLIB 命中结果
  - 去向：用于内部校验和日志

- `title`
  - 来源：LRCLIB 命中结果
  - 去向：用于内部校验和日志

- `duration_seconds`
  - 来源：LRCLIB 命中结果或查询输入回填
  - 去向：用于内部校验和日志

- `plain_lyrics`
  - 来源：LRCLIB 纯文本歌词
  - 去向：阶段一不进入下游，仅保留调试值

- `synced_lyrics`
  - 来源：LRCLIB 同步歌词原文
  - 去向：传给 LRC 解析器，构造 `lyric_sentence_units`

- `provider`
  - 来源：固定值 `lrclib`
  - 去向：仅用于调试与来源记录

- `provider_id`
  - 来源：LRCLIB 记录主键或等价标识
  - 去向：仅用于调试追踪

- `error`
  - 来源：异常信息或失败原因
  - 去向：仅用于日志和调试产物

#### 7.6 主链内部歌词标准化契约

无论歌词来自 LRCLIB 主链还是 FunASR 兜底链，进入算法层前都必须统一成现有 `perception.lyric_sentence_units` 可接受的结构。

阶段一统一结构如下：

```json
[
  {
    "start_time": 12.34,
    "end_time": 15.67,
    "text": "歌词内容",
    "confidence": 0.95,
    "token_units": [],
    "source_sentence_index": 0,
    "source": "lrclib|funasr"
  }
]
```

字段流转：

- `start_time`
  - LRCLIB 主链来源：LRC 行起始时间
  - FunASR 兜底来源：FunASR 分句后的起始时间
  - 去向：
    - 传给 `clean_lyric_units()`
    - 最终可能进入 `ModuleAOutput.lyric_units[].start_time`

- `end_time`
  - LRCLIB 主链来源：由“下一行起始时间”或“音频尾/可接受收尾规则”推导
  - FunASR 兜底来源：FunASR 分句后的结束时间
  - 去向：
    - 传给 `clean_lyric_units()`
    - 最终可能进入 `ModuleAOutput.lyric_units[].end_time`

- `text`
  - LRCLIB 主链来源：LRC 行文本
  - FunASR 兜底来源：FunASR 分句文本
  - 去向：
    - 传给 `clean_lyric_units()`
    - 最终进入 `ModuleAOutput.lyric_units[].text`

- `confidence`
  - LRCLIB 主链来源：规则默认值，阶段一由内部约定生成
  - FunASR 兜底来源：FunASR 置信度
  - 去向：
    - 传给 `clean_lyric_units()`
    - 最终进入 `ModuleAOutput.lyric_units[].confidence`

- `token_units`
  - LRCLIB 主链来源：阶段一默认空数组
  - FunASR 兜底来源：FunASR token 时间戳
  - 去向：
    - 传给 `clean_lyric_units()`
    - 若存在则进入 `attach_lyrics_to_segments()`
    - 最终可进入 `ModuleAOutput.lyric_units[].token_units`

- `source_sentence_index`
  - LRCLIB 主链来源：LRC 解析后的顺序索引
  - FunASR 兜底来源：FunASR 分句顺序索引
  - 去向：
    - 作为调试和挂载追踪字段
    - 最终可进入 `ModuleAOutput.lyric_units[].source_sentence_index`

- `source`
  - 来源：内部来源标记，值为 `lrclib` 或 `funasr`
  - 去向：阶段一建议仅保留在中间产物中，不直接写入当前 `ModuleAOutput` 最终契约

#### 7.7 算法层输入输出衔接

阶段一不改变现有算法层的主职责，只改变 `perception.lyric_sentence_units` 的来源。

当前衔接关系如下：

```json
{
  "perception.big_segments_stage1": "来自 allin1",
  "perception.beats": "来自 allin1",
  "perception.lyric_sentence_units": "来自 lrclib 或 funasr",
  "perception.sentence_split_stats": "来自 lrclib 解析统计或 funasr 分句统计"
}
```

字段流转：

- `perception.big_segments_stage1`
  - 来源：allin1 宏观段落输出
  - 去向：进入 `run_algorithm_stage()`，参与大段和小段计算

- `perception.beats`
  - 来源：allin1 beat 结构化输出
  - 去向：
    - 进入 `run_algorithm_stage()`
    - 最终进入 `ModuleAOutput.beats`

- `perception.lyric_sentence_units`
  - 来源：
    - 主链成功时来自 LRCLIB 解析结果
    - 主链失败时来自 FunASR
  - 去向：
    - 进入 `clean_lyric_units()`
    - 再进入 `attach_lyrics_to_segments()`

- `perception.sentence_split_stats`
  - 来源：
    - LRCLIB 主链时由 LRC 解析阶段生成基础统计
    - FunASR 兜底时复用现有 `sentence_split_stats`
  - 去向：
    - 进入内容角色编排流程
    - 作为窗口与歌词清洗的辅助统计

#### 7.8 最终 ModuleAOutput 契约映射

阶段一最终仍然输出现有 `ModuleAOutput`，字段映射关系如下。

```json
{
  "task_id": "RuntimeContext.task_id",
  "audio_path": "RuntimeContext.audio_path",
  "big_segments": "analysis_bundle.big_segments",
  "segments": "analysis_bundle.segments",
  "beats": "analysis_bundle.beats",
  "lyric_units": "analysis_bundle.lyric_units",
  "energy_features": "analysis_bundle.energy_features",
  "alias_map": "build_module_a_v2_alias_map(...)"
}
```

其中歌词相关字段必须满足以下最小契约：

```json
[
  {
    "segment_id": "string",
    "start_time": 12.34,
    "end_time": 15.67,
    "text": "歌词内容",
    "confidence": 0.95,
    "token_units": [],
    "source_sentence_index": 0,
    "unit_transform": "original|split|merged"
  }
]
```

字段流转：

- `segment_id`
  - 来源：`attach_lyrics_to_segments()` 根据时间重叠挂载到最终 `segments`
  - 去向：进入 `ModuleAOutput.lyric_units[].segment_id`

- `start_time`
  - 来源：主链或兜底链标准化结果，经清洗与必要裁剪后保留
  - 去向：进入 `ModuleAOutput.lyric_units[].start_time`

- `end_time`
  - 来源：主链或兜底链标准化结果，经清洗与必要裁剪后保留
  - 去向：进入 `ModuleAOutput.lyric_units[].end_time`

- `text`
  - 来源：清洗后的歌词文本
  - 去向：进入 `ModuleAOutput.lyric_units[].text`

- `confidence`
  - 来源：主链规则值或 FunASR 置信度，经清洗后保留
  - 去向：进入 `ModuleAOutput.lyric_units[].confidence`

- `token_units`
  - 来源：
    - LRCLIB 主链：阶段一通常为空
    - FunASR 兜底：沿用现有 token 时间戳
  - 去向：如存在则进入 `ModuleAOutput.lyric_units[].token_units`

- `source_sentence_index`
  - 来源：标准化分句顺序
  - 去向：如保留则进入 `ModuleAOutput.lyric_units[].source_sentence_index`

- `unit_transform`
  - 来源：歌词切分/合并/原样保留时的内部标记
  - 去向：如保留则进入 `ModuleAOutput.lyric_units[].unit_transform`

#### 7.9 阶段一新增中间产物建议

为保证可追溯性，阶段一建议新增以下中间产物。它们只服务于调试和恢复，不改变最终 `ModuleAOutput` 契约。

```json
{
  "artifacts/module_a_work_v2/perception/model/acoustid/fingerprint_result.json": "指纹生成标准化结果",
  "artifacts/module_a_work_v2/perception/model/acoustid/acoustid_match.json": "AcoustID 标准化命中结果",
  "artifacts/module_a_work_v2/perception/model/lrclib/lrclib_match.json": "LRCLIB 标准化查询结果",
  "artifacts/module_a_work_v2/perception/model/lrclib/lyric_sentence_units.json": "LRCLIB 解析后的内部歌词单元",
  "artifacts/module_a_work_v2/perception/model/lyrics/selected_provider.json": "本次实际采用 lrclib 或 funasr 的来源记录"
}
```

`selected_provider.json` 建议最小结构如下：

```json
{
  "provider": "lrclib|funasr",
  "reason": "synced_found|plain_only|not_found|identify_failed|query_failed|fallback",
  "task_id": "string"
}
```

这样可以直接回答三个问题：

- 本次任务最终用了哪条歌词链路；
- 为什么没有走 LRCLIB 主链；
- `ModuleAOutput.lyric_units` 最终来自哪里。
