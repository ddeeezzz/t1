Web 前端升级需求

## 一、技术方向

本次 Web 前端升级采用以下固定方案：

- 前端框架：Next.js App Router
- UI 组件库：Ant Design
- 数据层：TanStack Query
- 后端职责：保留现有 Python monitoring server 和现有 API，不由前端接管后端逻辑
- 模块 A 可视化：第一阶段继续使用 iframe 兼容现有 HTML 产物

说明：

- 当前目标不是重写整套后端，而是将现有监督页和任务操作能力迁移到成熟前端框架；
- Python 继续负责任务状态、任务调度、产物读取、WebSocket 推送和任务控制；
- Next.js 负责页面路由、组件组织、交互状态、任务工作台和审阅体验。

## 二、页面结构

### 1. 任务列表页

页面目标：

- 展示全部 task
- 支持查看任务状态
- 支持进入任务详情
- 支持创建 task
- 支持常见管理动作

核心区域：

- 顶部工具栏
  - 新建任务按钮
  - 搜索框
  - 状态筛选
  - 排序方式

- 任务表格区域
  - task_id
  - 任务名称
  - 当前状态
  - 当前模块进度
  - 创建时间
  - 更新时间
  - 操作列

- 操作列
  - 进入详情
  - 重命名
  - 复制任务
  - 重跑任务

### 2. 创建任务页

页面目标：

- 创建新的 pipeline task
- 提交音频路径或音频输入参数
- 填写可选提示信息

核心区域：

- 基础信息表单
  - 音频输入
  - task 名称
  - 可选歌词输入
  - 可选风格提示
  - 可选剧情提示

- 提交操作区
  - 创建任务按钮
  - 创建中状态
  - 创建失败提示

说明：

- 第一阶段表单字段以对齐现有后端创建接口为准；
- 不在前端额外发明新的业务字段。

### 3. 任务详情页

页面目标：

- 展示单个任务的全局状态
- 汇总当前任务可用产物
- 为监督、审阅、模块 A 可视化提供跳转入口

核心区域：

- 任务概览卡片
  - task_id
  - 名称
  - 输入音频
  - 当前状态
  - 当前模块
  - 创建时间
  - 更新时间

- 模块状态卡片
  - A / B / C / D / E 状态
  - done / running / failed / pending
  - 关键错误摘要

- 任务操作区
  - 重命名
  - 复制
  - 重跑

- 产物入口区
  - 进入监督页
  - 进入音画审阅页
  - 进入模块 A 可视化页

### 4. 任务监督页

页面目标：

- 替代现有 `task_monitor.html` 中的主监督能力
- 实时查看任务执行状态
- 查看 B/C/D 单元链路状态与错误摘要

核心区域：

- 模块状态概览
  - A / B / C / D / E 卡片
  - 当前状态
  - 开始时间
  - 完成时间
  - 失败信息摘要

- 单元链路表
  - unit_index
  - segment_id
  - shot_id
  - B 状态
  - C 状态
  - D 状态
  - 错误摘要

- 汇总统计区
  - done 数
  - running 数
  - failed 数
  - pending 数

说明：

- 本页优先复用现有 Python 监督接口和 WebSocket 推送；
- 不在第一阶段重写任务执行逻辑。

### 5. 音画审阅页

页面目标：

- 用于音画同步确认
- 对比视频与关键帧
- 查看 segment、歌词、scene、prompt 的对应关系

核心区域：

- 主播放器区
  - 视频播放
  - 音频进度
  - 当前时间
  - 总时长

- 当前段信息区
  - 当前 segment
  - 场景描述
  - 运镜信息
  - 当前段提示词入口

- Segment 滚动区
  - 展示全部 segment
  - 点击跳转时间点
  - 自动高亮当前 segment

- 歌词滚动区
  - 展示模块 A 输出的 `lyric_units`
  - 点击歌词跳转时间点
  - 自动高亮当前歌词

- 关键帧与提示词弹窗
  - 起始关键帧
  - 结束关键帧
  - 起始提示词
  - 结束提示词
  - 视频提示词

说明：

- 本页是第一阶段用户价值最高的页面之一；
- 页面目标不是营销展示，而是工作台式审阅工具。

### 6. 模块 A 可视化页

页面目标：

- 继续承载现有模块 A V2 HTML 可视化能力
- 在新前端框架下保留访问入口

核心区域：

- 顶部说明条
  - 当前文件来源
  - 打开新标签页入口

- iframe 展示区
  - 直接加载任务目录内模块 A 可视化 HTML

说明：

- 第一阶段不重写模块 A 可视化页面；
- 仅通过新前端进行壳层整合和导航统一。

## 三、路由结构

建议使用 Next.js App Router 组织以下路由：

### 1. 顶层路由

```text
/tasks
/tasks/create
/tasks/[taskId]
/tasks/[taskId]/monitor
/tasks/[taskId]/review
/tasks/[taskId]/module-a
```

### 2. 路由职责说明

- `/tasks`
  - 任务列表页
  - 展示全部任务和常见操作

- `/tasks/create`
  - 创建任务页
  - 提交创建表单

- `/tasks/[taskId]`
  - 任务详情页
  - 展示任务基础信息、模块状态和产物入口

- `/tasks/[taskId]/monitor`
  - 任务监督页
  - 连接 WebSocket，实时显示任务状态和链路表

- `/tasks/[taskId]/review`
  - 音画审阅页
  - 播放视频，滚动 segment 和歌词，查看关键帧与提示词

- `/tasks/[taskId]/module-a`
  - 模块 A 可视化页
  - 通过 iframe 展示现有模块 A HTML

### 3. Layout 结构建议

建议使用以下层级布局：

```text
app/
  tasks/
    layout.tsx
    page.tsx
    create/page.tsx
    [taskId]/
      layout.tsx
      page.tsx
      monitor/page.tsx
      review/page.tsx
      module-a/page.tsx
```

布局职责建议：

- `tasks/layout.tsx`
  - 全局导航
  - 页面头部
  - 面包屑容器

- `tasks/[taskId]/layout.tsx`
  - 单任务级别导航
  - 任务名称
  - Monitor / Review / Module A 页签切换

## 四、数据接口映射

前端第一阶段应直接对接现有 Python monitoring server 和现有 API。

### 1. 现有后端接口

根据当前服务端代码，已有接口包括：

```text
GET  /api/tasks
GET  /api/task
POST /api/task/create
POST /api/task/rename
POST /api/task/copy
POST /api/task/rerun
WS   /ws?task_id=...
GET  /web?task_id=...
```

说明：

- `GET /web` 是现有旧页面入口；
- 新前端迁移后，页面层将逐步由 Next.js 替代；
- Python 端继续保留这些业务接口和 WebSocket 推送能力。

### 2. 页面到接口的映射

#### 任务列表页

- 页面：`/tasks`
- 主要接口：
  - `GET /api/tasks`

用途：

- 获取任务列表
- 渲染任务表格
- 支持状态筛选、搜索、排序的前端逻辑

#### 创建任务页

- 页面：`/tasks/create`
- 主要接口：
  - `POST /api/task/create`

用途：

- 提交新任务创建请求
- 成功后跳转到对应任务详情页或监督页

#### 任务详情页

- 页面：`/tasks/[taskId]`
- 主要接口：
  - `GET /api/task?task_id=...`

用途：

- 获取任务详情
- 获取模块状态
- 获取当前产物路径与可视化入口

#### 任务监督页

- 页面：`/tasks/[taskId]/monitor`
- 主要接口：
  - `GET /api/task?task_id=...`
  - `WS /ws?task_id=...`

用途：

- 首次加载任务快照
- 接收任务实时状态更新
- 更新模块卡片、链路表与错误摘要

#### 音画审阅页

- 页面：`/tasks/[taskId]/review`
- 主要接口：
  - `GET /api/task?task_id=...`
  - 如现有详情接口已包含 review 所需字段，则第一阶段直接复用

用途：

- 获取视频路径
- 获取关键帧路径
- 获取 segment 时间轴
- 获取歌词时间轴
- 获取场景描述、prompt、camera 信息

说明：

- 若现有 `GET /api/task` 返回字段不足，第一阶段允许后端新增只读详情字段；
- 但不应让 Next.js 接管这些数据拼装逻辑。

#### 模块 A 可视化页

- 页面：`/tasks/[taskId]/module-a`
- 主要接口：
  - `GET /api/task?task_id=...`
  - 现有模块 A HTML 静态转发地址

用途：

- 获取 iframe 加载地址
- 获取模块 A 可视化文件是否存在

### 3. 前端数据层职责

TanStack Query 在第一阶段承担以下职责：

- 任务列表请求缓存
- 任务详情请求缓存
- 创建、复制、重命名、重跑等 mutation 状态管理
- 详情页与监督页的失效刷新
- WebSocket 更新后的数据回写或失效触发

说明：

- WebSocket 负责实时消息；
- Query 负责 HTTP 数据缓存与一致性；
- 不建议把所有状态都塞进本地组件状态。

## 五、迁移顺序

第一阶段迁移建议按“先搭新壳，再逐步替换旧页面能力”的顺序进行。

### 第一步：搭建前端框架骨架

目标：

- 初始化 Next.js App Router
- 接入 TypeScript
- 接入 Ant Design
- 接入 TanStack Query
- 建立基础 layout 和页面路由

交付结果：

- 可以访问 `/tasks`
- 可以访问 `/tasks/[taskId]`
- 可以访问空白版 monitor/review/module-a 页面

### 第二步：先完成任务列表与任务详情

目标：

- 打通 `GET /api/tasks`
- 打通 `GET /api/task`
- 打通 `POST /api/task/create`
- 补齐基础任务管理动作入口

交付结果：

- 可以创建 task
- 可以查看 task 列表
- 可以进入 task 详情
- 可以执行 rename / copy / rerun

说明：

- 这是第一阶段最基础的后台能力；
- 应优先落地。

### 第三步：迁移任务监督页

目标：

- 用新前端重建 monitor 页面
- 接入现有 WebSocket
- 替代现有 `task_monitor.html` 中的任务状态总览和链路表

交付结果：

- 模块状态卡片可实时刷新
- 单元链路表可实时更新
- 错误摘要可视化

说明：

- 这是现有功能的正式框架化迁移；
- 不改变后端执行语义。

### 第四步：迁移音画审阅页

目标：

- 重建播放器 + segment 滚动 + 歌词滚动 + prompt 对比能力
- 保留现有“点击跳转”和“自动高亮”的交互体验

交付结果：

- 可以播放视频
- 可以点击歌词和 segment 定位
- 可以查看当前段关键帧和提示词
- 可以做音画同步确认

说明：

- 这是第一阶段最核心的用户体验页面；
- 页面质量要求高于“只显示数据”。

### 第五步：接入模块 A 可视化页

目标：

- 在新前端中保留模块 A 可视化访问能力
- 通过 iframe 兼容现有 HTML 产物

交付结果：

- 可以从任务详情或导航进入模块 A 页
- 可以在新标签页打开模块 A HTML
- 可以在应用内直接查看模块 A 可视化

### 第六步：收口旧页面

目标：

- 在新页面覆盖主要能力后，逐步降低旧 `task_monitor.html` 的使用权重

交付结果：

- 新前端成为默认入口
- 旧页面保留为迁移期兼容入口

说明：

- 第一阶段不要求立即删除旧页面；
- 应先保证新页面覆盖核心功能，再考虑后续清理。
