# 命令组合与作用速查

本文专门解释你提到的几类命令如何组合使用：`run`、`resume`、`run-module`、`--force-module`、`--force`、单元级重试命令（`b-retry-segment/c-retry-shot/d-retry-shot/bcd-retry-segment`）。

目标是回答两个问题：

- 这条命令会重置哪些状态和产物索引？
- 执行后会自动推进到哪里，下一步该接什么命令？

## 1. 先记住这三个层级

- 全链路层：`run`、`resume`
- 单模块层：`run-module`
- 单元层：`b-retry-segment`、`c-retry-shot`、`d-retry-shot`、`bcd-retry-segment`

经验规则：越往下粒度越细，风险越小，但你需要手动补后续步骤的概率越高。

## 2. 一页对照表

| 命令 | 是否重置状态 | 自动执行范围 | 典型用途 | 最容易误解的点 |
|---|---|---|---|---|
| `run` | 不重置（除非加 `--force-module`） | 从 A 逻辑入口启动，A 后走 B/C/D 调度 | 新任务或整链重跑 | 不是“永远从头跑”，已 done 模块会被跳过 |
| `run --force-module X` | 重置 X 及下游 | 重置后继续执行全链路 | 从某模块开始“重建后续” | 这是全链路命令，不是只跑 X |
| `resume` | 不重置 | 从第一个非 done 模块继续 | 中断后续跑 | 若全部 done，会直接返回，不做事 |
| `resume --force-module X` | 重置 X 及下游 | 从 X 开始恢复 | 想“按恢复入口”强制改起点 | 和 `run --force-module` 的核心差异主要是入口语义，不是重置语义 |
| `run-module --module X` | 不重置 | 只执行 X 模块 | 调试单模块 | 不会自动跑下游 |
| `run-module --module X --force` | 重置 X 及下游 | 仍然只执行 X | 先清空再调试 X | 很多人以为会连下游一起跑，实际上不会 |
| `b-retry-segment` | 只重置目标 B 单元，然后把 C 起点及下游置为待重建 | 执行 B（定向） | 修一个 segment 的脚本/prompt | 不会自动重建 C/D |
| `c-retry-shot` | 只重置目标 C 单元 | 执行 C 后执行 D | 修一个关键帧并立即出新视频 | 前置要求 B 必须 done |
| `d-retry-shot` | 只重置目标 D 单元 | 只执行 D | 仅修最终视频段落 | 前置要求 C 必须 done |
| `bcd-retry-segment` | 重置目标链路的 B/C/D 单元 | 只补跑该链路并更新最终输出 | 跨模块链路定点修复 | 不是重跑所有 segment |

## 3. `force` 到底清空多大范围

以下是内部 `reset_from_module` 的实际行为（按代码）：

- 从 `A` 重置：模块状态重置 `A/B/C/D`，并删除 `B/C/D` 全部单元记录
- 从 `B` 重置：模块状态重置 `B/C/D`，并删除 `B/C/D` 全部单元记录
- 从 `C` 重置：模块状态重置 `C/D`，并删除 `C/D` 全部单元记录
- 从 `D` 重置：模块状态重置 `D`，并删除 `D` 全部单元记录

补充：任务级状态会被置为 `pending`，`output_video_path` 会清空。

## 4. 最常用的组合剧本（直接照抄）

### 剧本 A：任务中断后继续跑

```bash
uv run --no-sync mvpl resume --task-id <task_id> --config <config>
```

作用：从第一个非 done 模块继续。

### 剧本 B：我确认 C 之后都要重做

```bash
uv run --no-sync mvpl run --task-id <task_id> --config <config> --force-module C
```

作用：清空 C/D 并重建后续。

### 剧本 C：只想调试 B，不想立刻跑 C/D

```bash
uv run --no-sync mvpl run-module --task-id <task_id> --module B --force --config <config>
```

作用：清空 B/C/D，但只执行 B；适合反复改 prompt 或脚本。

后续要真正出视频，再接：

```bash
uv run --no-sync mvpl run --task-id <task_id> --config <config> --force-module C
```

### 剧本 D：只修一个 B segment，然后再补下游

```bash
uv run --no-sync mvpl b-retry-segment --task-id <task_id> --segment-id <seg_id> --config <config>
uv run --no-sync mvpl run --task-id <task_id> --config <config> --force-module C
```

作用：第一条只修 B 单元并把下游标为待重建；第二条重建 C/D。

### 剧本 E：只修一个 C shot，并立即出新视频

```bash
uv run --no-sync mvpl c-retry-shot --task-id <task_id> --shot-id <shot_id> --config <config>
```

作用：定向修 C，然后自动执行 D。

### 剧本 F：只修最终视频某个 D shot

```bash
uv run --no-sync mvpl d-retry-shot --task-id <task_id> --shot-id <shot_id> --config <config>
```

作用：只动 D，适合最终合成层问题。

### 剧本 G：跨模块只补一条链路

```bash
uv run --no-sync mvpl bcd-retry-segment --task-id <task_id> --segment-id <seg_id> --config <config>
```

作用：只重置并补跑该 `segment_id` 对应的 B/C/D 链路。

## 5. 你最容易踩的 4 个坑

- `run-module --force` 不会自动跑下游。它只执行你指定的模块。
- `b-retry-segment` 成功后也不会自动跑 C/D，这不是 bug，是设计。
- 不带 `--force-module` 的 `run` 可能会跳过已 done 模块，不是每次都全量重跑。
- 单元级重试命令会拒绝“同时存在其他非 done 单元”的场景，需要先把状态清干净。

## 6. 先查状态再下命令（推荐顺序）

- 先看 B：

```bash
uv run --no-sync mvpl b-task-status --task-id <task_id> --config <config>
```

- 再看 C：

```bash
uv run --no-sync mvpl c-task-status --task-id <task_id> --config <config>
```

- 再看 D：

```bash
uv run --no-sync mvpl d-task-status --task-id <task_id> --config <config>
```

- 看整条链路：

```bash
uv run --no-sync mvpl bcd-task-status --task-id <task_id> --config <config>
```

如果你只记一句话：

- 想“恢复进度”用 `resume`
- 想“强制从某模块重建”用 `run/resume --force-module`
- 想“只动一个模块”用 `run-module`
- 想“只修一个单元”用 `*-retry-*`

## 7. 新增：user_custom_prompt 注入规则（交互模式）

为避免把系统数据混进 User Prompt，模块 B 已改为：

- `System Prompt`：承载规则 + `input_payload_json`
- `User Prompt`：只承载 `user_custom_prompt`

交互式 CLI 中，当命令“可能触发 B”时会弹出询问：

- 是否添加本次 `user_custom_prompt`（默认否，等价空字符串）
- 若选择添加，可选三类来源：
  - 从 `configs/prompts/module_b_prompt.v*.md` 扫描到的 `user_prompt_template` 版本文本
  - 手动输入一句剧情/画风引导
  - 显式留空

触发该询问的命令：

- `run`、`resume`（若 `force-module` 选 `C/D` 则不会触发）
- `run-module --module B`（含 `--force`）
- `b-retry-segment`
- `bcd-retry-segment`

说明：此值只影响“本次请求”，不会回写配置文件。

## 8. 新增：任务日志文件命名规则

任务日志已改为“命令前缀 + 递增编号”，不再使用时间戳：

- `run_1.log`、`run_2.log`
- `resume_1.log`
- `run_module_b_1.log`

规则：

- 同一命令前缀独立计数
- 读取当前目录同名前缀日志，取最大数字 `+1`
- 非数字后缀文件（如 `run_abc.log`）会被忽略
