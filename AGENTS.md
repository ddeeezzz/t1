# AGENTS 指南：基于音频结构驱动的多模态音画同步生成系统

## 1. 文档目标与适用范围

本文件用于指导 coding agent 在本项目中的实现、重构、联调与评审工作，确保系统始终遵循以下主线：

- 结构优先：先建立可靠时间轴，再做视觉生成。
- 节拍驱动：所有关键切点必须受音频时间戳约束。
- 模块松耦合：模块间仅通过标准化 JSON 交换数据。
- 状态可恢复：所有阶段必须写入状态，支持断点续传。

默认实现范围为 `A -> B -> C -> D -> E` 的 MVP 核心链路。Web 前后端、视频动态化增强（AnimateDiff/SVD）属于可选扩展，不是首要阻塞项。

### 1.1 项目目录强约束（显式）

- 项目文件夹固定为：`t1/`（绝对路径：`m:/MyTest/working/zonghe/t1/`）。
- 所有业务代码、脚本、配置、文档与测试文件，均必须位于 `t1/` 内部。
- coding agent 在新增或修改文件时，默认工作根目录应视为 `t1/`，不得写入其外层目录。
- 若确需在 `t1/` 外放置文件，必须先在评审说明中给出必要性与风险评估，并获得明确批准。

## 2. 一句话目标、输入输出与成功标准

### 2.1 一句话目标

给定音频（可选歌词与用户辅助文本），自动生成与音乐结构和节拍高度对齐的连贯视频。

### 2.2 输入

- 必选：音频文件（如 `mp3/wav`）。
- 可选：歌词文本或歌词时间戳信息。
- 可选：用户风格提示、剧情提示、视觉限制条件。

### 2.3 输出

- 结构化中间数据（模块 A/B 输出 JSON）。
- 中间图像或片段（模块 C）。
- 最终成片（模块 D 输出 `mp4`）。
- 任务状态记录（模块 E，SQLite）。

### 2.4 成功标准

- 视频切点与节拍时间轴严格对齐，不允许随意偏移关键卡点。
- 场景切换遵循音乐宏观段落边界。
- 全链路可中断恢复，不重跑已完成模块。
- 关键中间结果具备可追溯性（可定位输入、输出、状态）。

## 3. 架构基线与实现边界

### 3.1 技术栈基线（默认）

- 核心编排与算法实现：Python。
- 状态存储：SQLite。
- 视频后处理：FFmpeg（可由 Python 调用）。
- 图像生成：Diffusers + Stable Diffusion（1.5/XL 可选）。
- 语音与歌词链路：Demucs + Whisper。
- 音乐结构链路：Librosa + Allin1。

### 3.2 非默认但允许的扩展

- 动态化增强：AnimateDiff/SVD（可选）。
- Web 产品化前后端（可选）。

扩展能力不得破坏 MVP 核心链路与接口契约。

### 3.3 模块职责边界（固定）

- 模块 A（音乐理解）：音频解析、双链融合、时间轴输出。
- 模块 B（视觉脚本）：根据 A 输出生成分镜脚本 JSON。
- 模块 C（图像生成）：根据 B 输出生成关键帧与风格一致性结果。
- 模块 D（视频合成）：按绝对时间轴拼接视频并混音导出。
- 模块 E（状态管理）：记录任务状态，驱动断点续传。

任何模块都不得隐式承担上游核心职责，不得越权改写上游确定的关键时间戳。

## 4. Pipeline 执行顺序与状态机规则

### 4.1 执行顺序

1. 初始化任务（E）：写入任务元信息，状态 `pending`。
2. 执行 A：产出 `ModuleAOutput`，状态更新为 `done` 或 `failed`。
3. 执行 B：仅当 A=`done` 才可启动，产出 `ModuleBOutput`。
4. 执行 C：仅当 B=`done` 才可启动，产出关键帧资源。
5. 执行 D：仅当 C=`done` 才可启动，输出最终视频。
6. 收尾写库（E）：更新全链路结果路径与终态。

### 4.2 TaskState（SQLite 枚举）

- `pending`：等待执行。
- `running`：正在执行。
- `done`：成功完成。
- `failed`：失败，可重试。

### 4.3 断点续传规则

- 模块开始前必须检查上游是否 `done`。
- 模块执行中必须及时写入 `running`。
- 模块结束必须写入 `done/failed` 与输出路径。
- 重启后仅从第一个非 `done` 模块恢复，禁止重跑已完成模块。

### 4.4 失败恢复规则

- 外部依赖失败（模型/API/IO）后，允许单模块重试。
- 下游模块失败不得回滚上游 `done` 状态。
- 可选模块失败时可降级，但必须记录降级策略和原因。

## 5. MVP 优先级与降级策略

### 5.1 默认优先级

1. 优先保证 A 与 C 可端到端联通（先有稳定时间轴与可见结果）。
2. 其次保证 B 的结构化分镜输出稳定。
3. 再保证 D 的高精度拼接与转场策略。
4. E 全程必需，不可降级为“无状态运行”。

### 5.2 可接受降级

- A：歌词链路不可用时，降级为纯声学结构链，但必须明确标记。
- B：LLM 质量波动时，降级为模板化脚本生成。
- C：显存不足时，降级分辨率或模型版本。
- D：动态化增强失败时，降级为静态关键帧拼接。

## 7. Coding Agent 执行规范

### 7.1 实现与变更原则

- 先保证契约正确，再优化算法效果。
- 任何改动必须保持模块边界清晰。
- 若改动影响上下游字段，必须同步更新文档与兼容逻辑。
- 不得绕过模块 E 直接执行“无状态流水线”。

### 7.2 提交前最小自检

1. 是否破坏了 `ModuleAOutput/ModuleBOutput` 最低字段。
2. 是否存在下游在上游非 `done` 时被触发的情况。
3. 是否保留了时间戳硬约束。
4. 是否写入了完整状态与输出路径。

### 7.3 Git 协作约定（用户主导）

- 本项目所有 Git 操作默认由用户自行执行，coding agent 不主动执行 Git 命令。
- 仅当用户显式指定“执行某个具体 git 命令”时，coding agent 才可执行该命令。
- 即使被授权执行 Git 命令，也仅执行用户明确指定的命令，不扩展执行其他 Git 操作。
- 当需要提交代码时，`commit message` 默认使用中文，建议采用“类型(范围): 中文摘要”的格式（如：`feat(module_a): 修复歌词节拍吸附阈值判断`）。

## 9. PR/Review 检查清单

评审时至少逐项检查以下内容：

1. 是否遵守中文注释与中文日志要求。
2. 每个文件是否包含文件头注释。
3. 每个函数是否包含中文注释（含参数、返回、异常/边界）。
4. 每个导入是否包含用途注释。
5. 是否遵守模块边界与执行顺序。
6. 是否遵守 `pending/running/done/failed` 状态写入规则。
7. 是否保持时间戳硬约束，不随意偏移关键切点。
8. 是否存在未文档化字段在模块间透传。
9. 是否提供了必要的降级记录与恢复说明。

## 10. 禁止项

- 禁止绕过时间戳硬约束进行“目测对齐”。
- 禁止跳过状态写入直接驱动下游模块。
- 禁止在模块间传递未文档化字段。
- 禁止以下游逻辑反向篡改上游已确认时间轴。
- 禁止将可选增强能力作为 MVP 必须依赖。
- 禁止在 `t1/` 目录之外新增或修改项目代码文件（经明确批准的例外场景除外）。
- 禁止主动执行任何 Git 命令；除非用户显式指定要执行的具体 git 命令。

## 11. 与开题报告冲突时的决策优先级

若实现细节与开题报告存在冲突，默认按以下优先级决策：

1. 结构优先。
2. 节拍驱动。
3. 模块松耦合。
4. 状态可恢复。

所有偏离必须在 PR 描述中给出原因、风险与回退方案。
